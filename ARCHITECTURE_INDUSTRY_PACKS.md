# Industry Packs — Архитектура коробочных версий Staffix

**Статус:** Драфт, разработка в ветке `industry-packs` (от main `1866bb1`).
**Дата старта:** 8 мая 2026.
**Первая коробка:** Туроператор (полное покрытие 10/10).
**Решение по слиянию:** ветка живёт изолированно, единый squash-merge в main только после полной обкатки на живом туроператоре (Фаррух).

---

## 1. Принципы

### 1.1. Почему по индустрии, а не по service/sales

Сейчас Staffix делит бизнесы на `dashboardMode: "service" | "sales"` — это первичная ось продукта. Это слишком грубо: туроператор и салон оба «service», но их потребности (терминология, поля карточки клиента, автоматизации, дашборд) несовместимы.

**Новая ось — индустрия.** Это более точная единица деления. Один пользователь — одна индустрия — одна коробка.
`dashboardMode` остаётся на низком уровне, но **становится производным** от коробки (туроператор — гибрид: и продаёт, и записывает).

### 1.2. Цель коробки

Туроператор регистрируется и видит **«Staffix для туроператоров»**, а не «универсальный Staffix с настройками». В первом окне после онбординга:
- Дашборд туроператора, не общий
- Каталог уже наполнен примерами туров (10-20 штук)
- FAQ закрыты (визы, страховка, перенос, отмена)
- AI-бот говорит как туроператор с первого сообщения
- Поля карточки клиента туроператорские (паспорт, виза, состав семьи, бюджет)
- Автоматизации запущены (напоминания о вылете, follow-up после возвращения)
- Терминология: «тур», «бронь», «вылет» вместо «услуга», «запись», «дата визита»

Пользователю остаётся только подкрасить под свой реальный каталог. **Это и есть коробка.**

### 1.3. Чего НЕ делаем в этой итерации

- Не строим 21 индустрию параллельно. Туроператор — пилот 10/10. Остальные 20 — после.
- Не ломаем существующих пользователей. Они работают как раньше, пока сами не нажмут «Обновить до коробочной версии».
- Не убираем `dashboardMode` сразу. Он остаётся, но становится derived field от коробки.
- Не делаем миграцию данных существующих бизнесов автоматически.
- Не добавляем биллинг/Premium tier для коробок на старте — они бесплатны во всех тарифах.

## 2. Путь клиента (UX-сценарий)

### 2.1. Новый туроператор (Алишер регистрируется)

**Шаг 1 — регистрация (без изменений).**
Имя, телефон, email, пароль.

**Шаг 2 — выбор индустрии (первичный экран онбординга).**
Один выбор, не множественный. Карточки крупные, с иконкой и кратким описанием.

```
Какая у вас индустрия?

[✈️ Туроператор]      [🍽️ Ресторан]        [💇 Салон красоты]
[🏥 Клиника]          [🚗 Автосервис]       [🏠 Недвижимость]
... (все 21)
[🏢 Другое — расскажите]
```

При выборе «Другое» — текстовое поле «Опишите ваш бизнес» → ставится `installedPackId = null`, дальше идёт generic-онбординг.

**Шаг 3 — уточнения, специфичные для индустрии (sub-config).**

Для туроператора:
```
Расскажите о вашем бизнесе:
- Направления работы (множественный выбор):
  [☐ Турция] [☐ ОАЭ] [☐ СНГ] [☐ Внутренний] [☐ Премиум] [☐ Виза] [☐ Бизнес-туризм]
- Формат работы:
  ( ) Розница (продаём конечному клиенту)
  ( ) B2B (работаем через агентов)
  ( ) Гибрид
- Средний чек тура:
  ( ) до 500$  ( ) 500-1500$  ( ) 1500-5000$  ( ) от 5000$
```

Эти ответы сохраняются в `Business.packConfig` как JSON и влияют на содержимое seeders (например, если выбраны только Турция и ОАЭ, в каталог не льются примеры туров по СНГ).

**Шаг 4 — финальное «Начать работу».**
Спиннер 5-10 секунд: *«Настраиваем для вашей индустрии…»*. За это время `applyPack(businessId, "travel", "fresh", packConfig)`:
1. Создаёт записи в `Service` (10-20 примеров туров)
2. Создаёт записи в `FAQ` (15-20 готовых)
3. Создаёт записи в `AutomationSettings` / `AdminAutomation` (5-7 туроператорских)
4. Заполняет `Business.clientFieldsConfig` (паспорт, виза, состав семьи, бюджет, направление мечты)
5. Включает туроператорский `Business.systemPrompt`
6. Прописывает `Business.welcomeMessage`
7. Сохраняет `Business.installedPackId = "travel"`, `Business.packConfig`, `Business.packAppliedAt`
8. Создаёт аудит-запись в `IndustryPackInstall`

**Шаг 5 — первый дашборд.**
Алишер видит туроператорский дашборд (не generic):

| Виджет | Что показывает |
|---|---|
| Активные брони | количество, ближайшие вылеты |
| Календарь вылетов | следующие 14 дней |
| Сезонная нагрузка | график бронирований по месяцам |
| Топ направлений | какие туры лучше продаются |
| Туры в продаже | каталог с количеством бронирований по каждому |
| Заявки на подбор | новые лиды от AI-бота |
| AI-инсайты | стандартный виджет (общий для всех индустрий) |

Навигация:
```
Главная | Туры | Брони | Клиенты | Направления | Чаты | Команда | Настройки
```

(вместо стандартного: Главная | Услуги | Записи | Клиенты | Чаты | …)

**Шаг 6 — пользователь начинает работу.**
- Открывает «Туры» → видит 15 примеров с реальными ценами и описаниями
- Удаляет нерелевантные, переименовывает свои, добавляет свои
- Загружает PDF-описания туров в базу знаний
- Подключает Telegram-бота
- Готов к работе

### 2.2. Существующий пользователь (легаси)

При входе в дашборд видит баннер:
```
🎁 Доступна коробочная версия для вашей индустрии: «Туроператор»
Получите готовый каталог, FAQ, автоматизации и улучшенный дашборд.
Ваши текущие данные останутся нетронутыми.
                                   [Подробнее] [Обновить]
```

Нажимает «Подробнее» → модалка:
```
Что добавится:
✅ 15 примеров туров в каталог (вы можете удалить)
✅ 18 готовых FAQ
✅ 6 автоматизаций
✅ 5 новых полей в карточке клиента (паспорт, виза, …)
✅ Туроператорский AI-промпт
✅ Виджеты дашборда (календарь вылетов, сезонная нагрузка, топ направлений)
✅ Терминология: «услуга» → «тур», «запись» → «бронь»

Что НЕ изменится:
🔒 Ваши существующие услуги/туры
🔒 Ваши клиенты и брони
🔒 Ваша база знаний
🔒 Ваши настройки бота и каналы

                              [Отмена] [Обновить до коробки]
```

При подтверждении: `applyPack(businessId, "travel", "merge", packConfig)`. Режим `merge` — НИКОГДА не перезаписывает существующее.

## 4. Модель данных (изменения в Prisma)

### 4.1. Изменения в `Business`

Добавляются:
```prisma
installedPackId  String?    // "travel", "restaurant", null для legacy/other
packConfig       Json?      // ответы из step 3 онбординга
packAppliedAt    DateTime?  // последняя дата apply
packVersion      Int?       // версия пакета на момент apply (для будущих обновлений)
```

`dashboardMode` остаётся, но становится **производным**: если `installedPackId` установлен — берётся `manifest.derivedDashboardMode`. Если нет — используется явное значение (legacy).

### 4.2. Новая модель `IndustryPackInstall`

Аудит каждого `applyPack`:
```prisma
model IndustryPackInstall {
  id           String   @id @default(cuid())
  businessId   String
  business     Business @relation(...)
  packId       String
  packVersion  Int
  mode         String   // "fresh" | "merge"
  packConfig   Json
  servicesCreated     Int @default(0)
  faqsCreated         Int @default(0)
  automationsCreated  Int @default(0)
  clientFieldsAdded   Int @default(0)
  promptsApplied      Boolean @default(false)
  appliedAt    DateTime @default(now())
  appliedBy    String?  // userId, кто инициировал
}
```

Используется для:
- Понимания истории изменений (когда юзер обновил коробку, когда переустановил)
- Отката (если что-то пошло не так — мы знаем что было создано)
- Аналитики (сколько бизнесов установили каждую коробку)

### 4.3. Что НЕ меняется

- `Service`, `FAQ`, `AutomationSettings`, `Client` — таблицы остаются нейтральными, никаких индустриальных полей в них.
- Существующие записи существующих пользователей не трогаются никогда. `applyPack(merge)` только добавляет, никогда не редактирует и не удаляет.

## 5. Терминология — `useTerminology`

### 5.1. Принцип

**Код всегда говорит нейтрально:** `service`, `booking`, `appointment`, `client`. Эти имена живут в БД, в API, в коде.

**UI всегда говорит индустриально:** «Тур», «Бронь», «Вылет», «Турист». Эти слова берутся из `manifest.terminology` активной коробки.

Перевод происходит на стыке: компонент UI вызывает `useTerminology()` и подставляет нужное слово.

### 5.2. Хук

```typescript
// в каком-нибудь src/lib/industry-packs/hooks/useTerminology.ts

export function useTerminology() {
  const business = useCurrentBusiness();
  const pack = business.installedPackId ? getManifest(business.installedPackId) : null;
  const dict = pack?.terminology ?? defaultTerminology;

  return (key: TerminologyKey, form: "singular" | "plural" = "singular") => {
    return dict[key]?.[form] ?? defaultTerminology[key][form];
  };
}
```

### 5.3. Применение

В существующих UI-компонентах меняем хардкод на хук:

**Было:**
```tsx
<h1>Услуги</h1>
<button>Создать услугу</button>
```

**Стало:**
```tsx
const t = useTerminology();
<h1>{t("service", "plural")}</h1>
<button>Создать {t("service")}</button>
```

Туроператор видит «Туры», «Создать тур». Салон видит «Услуги», «Создать услугу». Один компонент, два UI.

### 5.4. Дефолтный словарь

Когда коробка не установлена (легаси, «другое») — используются дефолтные термины. Это те же слова что сейчас в коде:
```typescript
const defaultTerminology = {
  service: { singular: "Услуга", plural: "Услуги" },
  booking: { singular: "Запись", plural: "Записи" },
  appointment: { singular: "Запись", plural: "Записи" },
  client: { singular: "Клиент", plural: "Клиенты" },
};
```

### 5.5. Что остаётся нейтральным

- Имена переменных в коде
- Имена полей в БД
- API endpoints (`/api/services`, `/api/bookings`)
- Вебхуки и интеграции
- Логи и аналитика

```tsx
function Dashboard() {
  const layout = useDashboardLayout(); // ["active-bookings", "flight-calendar", ...]
  return (
    <Grid>
      {layout.map(widgetId => {
        const Component = widgetRegistry[widgetId];
        return Component ? <Component key={widgetId} /> : null;

```

Если коробка ссылается на неизвестный виджет — он молча пропускается (никаких 500). Логируется warning.


## 7. Жизненный цикл `applyPack`

### 7.1. Сигнатура

```typescript
async function applyPack(
  businessId: string,
  packId: string,
  mode: "fresh" | "merge",
  packConfig: Record<string, unknown>,
  initiatedBy: string  // userId
): Promise<IndustryPackInstallResult>
```

### 7.2. Шаги исполнения

1. **Загрузка**: получаем manifest, seeders, prompts из реестра по `packId`. Если `packId` не найден — ошибка.
2. **Транзакция Prisma** (всё или ничего):
   1. Обновляем `Business`: `installedPackId`, `packConfig`, `packAppliedAt`, `packVersion`, `systemPrompt`, `welcomeMessage`, `dashboardMode` (= manifest.derivedDashboardMode).
   2. **Services** (`mode == "fresh"` — все из seeders; `mode == "merge"` — только если у бизнеса нет ни одной услуги).
   3. **FAQs**: аналогично (в merge — только если нет FAQ).
   4. **Automations**: аналогично.
   5. **Client fields config** (`mode == "fresh"` — replace массива; `mode == "merge"` — добавить ключи которых ещё нет).
   6. Создаём запись в `IndustryPackInstall` с подсчётами.
3. **Возвращаем** результат: что создано, что пропущено.

### 7.3. Идемпотентность

Повторный вызов `applyPack` на бизнесе с уже установленной коробкой:
- В `fresh` режиме — **НЕ ДОПУСКАЕТСЯ** (защита от перезаписи). Только при явном флаге `forceFresh: true` (для админов).
- В `merge` режиме — **ВСЕГДА БЕЗОПАСЕН**. При повторном merge ничего не дублируется.

### 7.4. Откат

Откат полноценный не делаем (сложно — пользователь мог уже добавить свои данные). Вместо этого:
- В `IndustryPackInstall` есть полный аудит: что было создано
- Админ может посмотреть и удалить вручную
- В UI пользователя есть «Удалить пример тура» по каждой записи коробки (через флаг `Service.fromPack: boolean`)

## 8. Миграция существующих пользователей

### 8.1. Этапы внедрения для legacy

**Этап 1 (с самого начала):** новые пользователи проходят через flow с выбором индустрии. Существующие — НЕ ТРОГАЮТСЯ. У них `installedPackId = null`, всё работает как было.

**Этап 2 (когда туроператорская коробка обкатана):** в дашборде legacy-пользователей с `businessType == "travel"` появляется баннер «Доступна коробка». Кнопка «Обновить» вызывает `applyPack(merge)`.

**Этап 3 (когда есть несколько коробок):** баннеры появляются для всех совпадающих индустрий.

**Этап 4 (опциональный, не сейчас):** для пользователей которые НЕ нажали «Обновить» — массовая авто-миграция в merge режиме (не перезапишет ничего, только дополнит). Решение об этом принимается отдельно.

### 8.2. Защита от потери данных

`mode == "merge"` — единственный путь для legacy. Гарантии:
- Услуги не дублируются и не создаются если у бизнеса уже есть хотя бы одна
- FAQ не создаются если уже есть
- Автоматизации не создаются если уже есть
- `clientFieldsConfig` — только добавление, никогда удаление существующих
- `systemPrompt`, `welcomeMessage` — **спрашиваем перед перезаписью** (модалка «у вас уже есть свой промпт, заменить или оставить?»)

### 8.3. Коммуникация

После применения коробки — toast/модалка:
```
✅ Коробка туроператора установлена
Добавлено:
- 15 примеров туров
- 18 FAQ
- 6 автоматизаций
- 5 полей карточки клиента
Терминология обновлена: «услуга» → «тур», «запись» → «бронь»
                                              [Закрыть] [Посмотреть]
```

## 9. Edge cases

### 9.1. Пользователь выбрал «Другое» при онбординге

`installedPackId = null`. Пользователь работает в legacy режиме (нейтральная терминология, дефолтный дашборд). Он сохраняется и работает, но без коробки.

### 9.2. Пользователь хочет сменить индустрию после установки

Сценарий: установил «Туроператор», через месяц понял что он скорее «Турагентство + B2B». Допустим (тот же `packId = travel`, но другой `packConfig`).

Решение: даём кнопку «Изменить настройки коробки» в Settings. Это **повторный merge** — добавит то чего ещё нет, ничего не сломает.

Сценарий жёстче: установил «Туроператор», понял что он на самом деле «Ресторан» (выбрал не то). Это **смена коробки** — даём кнопку «Сменить индустрию» с предупреждением: «Услуги и FAQ останутся, но дашборд и терминология полностью поменяются». При подтверждении — `applyPack("restaurant", "merge")`.

### 9.3. Коробка обновлена централизованно

Мы доработали туроператорскую коробку — добавили 3 новых FAQ и 1 виджет. Что с уже установившими?
- **Не делаем авто-обновление.** Пользователь увидит баннер «Коробка туроператора обновлена. Доступно: 3 новых FAQ, 1 виджет. Применить?»
- При нажатии — `applyPack(merge)` с новой версией.

`packVersion` в БД позволяет точно знать кто на какой версии и что им предложить.

### 9.4. Пользователь удалил все примеры туров и теперь fresh-применил коробку повторно

Защита: `applyPack(fresh, forceFresh: true)` доступен только админам через support-tools. Юзеру в UI — только merge.

Если admin делает forceFresh — всё содержимое коробки заливается заново. Существующие данные ОСТАЮТСЯ (не удаляются), просто добавляются новые. Возможны дубликаты — это осознанный outcome (admin принял решение).

### 9.5. Бизнес с несколькими каналами (Telegram + Instagram + WhatsApp)

Коробка применяется на уровне `Business`, не на уровне канала. Все каналы получают одни и те же AI-промпт, FAQ, услуги. Это правильно — бизнес один, индустрия одна.

## 10. Чек-лист: как добавить новую индустрию

После того как туроператорская коробка готова и движок работает, добавление любой новой индустрии — это:

1. **Создать папку** `src/lib/industry-packs/<name>/`
2. **Написать manifest.ts** (id, name, icon, terminology, dashboardLayout, navigation, derivedDashboardMode, subConfig)
3. **Написать seeders.ts** (services, faqs, automations, clientFields)
4. **Написать prompts.ts** (systemPrompt, welcomeMessage)
5. **Опционально написать components/** (если нужны индустриальные виджеты — например, для ресторана `<TableMap>`, для автосервиса `<VinDecoder>`)
6. **Опционально написать api/** (если нужны индустриальные endpoints)
7. **Зарегистрировать** в `src/lib/industry-packs/registry.ts`
8. **Зарегистрировать виджеты** в `src/lib/industry-packs/widgets/registry.ts`
9. **Зарегистрировать пункты навигации** в `src/lib/industry-packs/navigation/registry.ts` (если новые)
10. **Тест fresh-установки**: создать тестовый бизнес → онбординг с этой индустрией → проверить что всё применилось правильно
11. **Тест merge-установки**: применить к legacy-бизнесу → проверить что ничего не перезаписалось
12. **Тест unlearn**: удалить коробку (если будем поддерживать) — проверить что данные пользователя остались

Целевое время добавления одной новой индустрии после готовности движка: **3-5 дней работы**.

## 11. Этапность для туроператорской коробки

Разбиение на итерации. Каждая итерация — самостоятельный набор коммитов в ветке `industry-packs`. После каждой — пишем заметку в этом документе про статус.

### Итерация 1 — Каркас движка (без контента)
- Prisma changes: добавить `installedPackId`, `packConfig`, `packAppliedAt`, `packVersion` в `Business`; новая модель `IndustryPackInstall`.
- Миграция Prisma `20260601_industry_packs_skeleton`.
- `src/lib/industry-packs/types.ts` — типы манифестов и сидеров.
- `src/lib/industry-packs/registry.ts` — пустой реестр (пока без коробок).
- `src/lib/industry-packs/apply-pack.ts` — реализация applyPack (fresh + merge).
- `src/lib/industry-packs/hooks/useTerminology.ts`, `useInstalledPack.ts`, `useDashboardLayout.ts`, `useNavigation.ts`.
- `src/lib/industry-packs/widgets/registry.ts` — пустой реестр виджетов.
- Дефолтный словарь терминологии.
- **Не подключаем к UI пока.** Просто инфраструктура.

### Итерация 2 — Туроператорская коробка (контент)
- `src/lib/industry-packs/travel/manifest.ts` — полный манифест.
- `src/lib/industry-packs/travel/seeders.ts` — 15 примеров туров, 18 FAQ, 6 автоматизаций, 5 полей клиента.
- `src/lib/industry-packs/travel/prompts.ts` — systemPrompt переехал из prompt-templates.ts, welcomeMessage.
- Регистрация в реестре.

### Итерация 3 — Туроператорские виджеты дашборда
- `<FlightCalendar>` — календарь вылетов на 14 дней
- `<SeasonLoadWidget>` — график сезонной нагрузки
- `<ActiveBookingsWidget>` — счётчик активных бронирований
- `<TopDestinationsWidget>` — топ направлений месяца
- `<ToursCatalogSummary>` — сводка по каталогу
- Регистрация в widget registry.

### Итерация 4 — Динамический дашборд и навигация
- Переписываем `src/app/dashboard/page.tsx` (или layout) на использование `useDashboardLayout`.
- Переписываем sidebar-навигацию на `useNavigation`.
- Подключаем `useTerminology` в основные UI-компоненты (Service list, Booking list, Client form, …).

### Итерация 5 — Новый онбординг flow
- `src/app/onboarding/page.tsx` — переписать на: step 1 (как сейчас) → step 2 (выбор индустрии) → step 3 (sub-config) → step 4 (apply).
- Спиннер «Настраиваем для вашей индустрии…».
- API роут `/api/onboarding/apply-pack`.

### Итерация 6 — Тест на Фаррухе
- Создаём для Фарруха тестовый бизнес «Туроператор».
- Проводим через новый онбординг.
- Собираем фидбэк.
- Правим что не так.

### Итерация 7 — Полировка туроператорской коробки
- Дорабатываем по фидбэку от Фарруха.
- Добавляем то что забыли (industry-specific поля Tour: длительность тура, страна, категория отеля, питание).
- Может появиться отдельная таблица `Destination` если нужно управление списком направлений.

### Итерация 8 — Миграция существующих пользователей
- Баннер «Доступна коробка» в дашборде для legacy-пользователей с `businessType == "travel"`.
- Модалка «Что добавится» / «Что не изменится».
- Кнопка «Обновить до коробки» → `applyPack(merge)`.

### Итерация 9 — Финальный squash-merge в main
- Полный QA на ветке.
- Миграции Prisma собираются в одну (или несколько) последовательных.
- Squash-merge одним PR с подробным описанием.
- Деплой.
- Мониторинг 3-7 дней на проде.

## 12. Открытые вопросы и риски

### 12.1. Открытые вопросы (требуют решения по ходу)

- **Tour как отдельная модель или Service с industry-полями?** Скорее всего — `Service` остаётся, добавляем опциональные industry-поля (например, `Service.industryFields: Json` для длительности тура, страны, категории отеля). Решим в Итерации 7 по фидбэку.
- **Управление списком направлений** — отдельная таблица `Destination` или просто справочник? Зависит от того, как туроператоры реально работают. Спросим Фарруха.
- **Перевод существующих туроператоров (которые сейчас выбрали `travel` в businessType)** — нужно ли в Итерации 6 авто-предложить им коробку? Скорее всего да, но с явным согласием.
- **Локализация коробок (узбекский, казахский)** — коробка туроператора должна быть на ru/uz/kz? Скорее всего да, но это +30% работы. Решим после готовности RU-версии.

### 12.2. Риски

- **Размер ветки**: к моменту merge ветка может содержать 30-50 коммитов и 5-7 миграций Prisma. Чем дольше живёт ветка — тем больнее merge. Митигация: регулярный (раз в неделю) merge from main → ветка.
- **Конфликты с операционными правками на main**: на main активно идёт работа (PayPro, partner program, AI smart routing, activity log). Любая правка в `Business`, `Service`, `FAQ`, `AutomationSettings`, `prompt-templates.ts` создаёт потенциальный конфликт. Митигация: прошу избегать масштабных рефакторов этих файлов на main пока ветка коробок жива.
- **Сломаем существующих пользователей**: если в коде есть невидимые ассампшны вокруг `dashboardMode` или `businessType`, переход на `installedPackId` может что-то сломать. Митигация: оставляем `dashboardMode` как derived field, не удаляем; делаем явные fallback во всех новых UI-компонентах для случая `installedPackId = null`.
- **Manifest growth**: со временем каждый manifest может разрастись до 1000+ строк (21 индустрия × 1000 = 21000 строк только манифестов). Митигация: разбивать manifest на модули (`manifest.ts`, `terminology.ts`, `dashboardLayout.ts`) внутри папки коробки если нужно.
